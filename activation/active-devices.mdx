<p className="bf-lead">
Active device is the public pricing word for a device, server, or runtime identity that actually triggered Bitfield during the billing window.
</p>

It does not mean "installed once forever." It does not mean "authorized in theory." It does not mean "how many seconds the runtime ran." It means the runtime path was triggered for that identity in the billing window.

## What counts

| Runtime identity | Counts when |
|---|---|
| Laptop or desktop | Bitfield is triggered there during the billing window. |
| Phone or tablet | Bitfield is triggered there during the billing window. |
| Server, cloud runtime, or local box | Bitfield is triggered there during the billing window. |
| Test machine or CI runner | Bitfield is triggered there during the billing window. |
| Customer device | That customer environment triggers Bitfield runtime access. |

## What does not count by itself

| Thing | Why |
|---|---|
| Old installed files | Dormant files are not a trigger. |
| Public website visitor | Viewing output is not runtime access. |
| Static exported page | Visitors reading static files do not trigger Bitfield. |
| Same runtime doing lots of work | Usage amount is not the unit. |
| Package folder | A package is product material, not a runtime trigger. |

## Local activation state

Runtime Kit package state and engine storage live under `~/.bitfield/`, and activation has its own sibling lane for this device's signed permission. That local state is not something the customer should edit by hand.

| Local idea | Customer rule |
|---|---|
| Device key material | Created by the activation flow, not invented manually |
| Signed permission | Comes from account-owned activation |
| Old blocked permissions | Remembered so revoked access does not silently return |
| Runtime Kit package state | Separate from activation; do not mix the two |

## Common failures

| Symptom | Likely cause | Fix |
|---|---|---|
| New computer cannot trigger Bitfield | Device path was not activated yet | Use the account portal activation flow |
| Old computer still appears active | Runtime identity was not revoked or replaced | Manage active devices in the account portal |
| User copies one device's local activation folder to another | Local permission was treated like a portable key | Activate the second runtime identity properly |
| Package tooling tries to own activation | Runtime Kit and activation boundaries got mixed | Keep activation in the account/key lane |
| Billing is explained as runtime seconds | Trigger count and runtime amount were mixed up | Say one triggered identity per window, not per second |

## Verify

The active-device model is clear when the user can tell the difference between:

- the plan subscription
- a triggered runtime/device unit
- local activation state
- Runtime Kit package state
- public static traffic
