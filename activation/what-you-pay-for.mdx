Bitfield billing has two pieces: your plan subscription, and the runtime/device units that get counted when Bitfield is actually triggered.

That word matters: **triggered**. A dormant install is not the same thing as a triggered runtime. A visitor looking at static output is not the same thing as a triggered runtime. A million clicks on the same already-triggered runtime are not a million runtime/device units.

The public pricing word you will see is active device. It means a device, server, or runtime identity that triggered the Bitfield runtime grant path during the billing window.

## The short version

| Situation | What happens |
|---|---|
| You keep a paid plan after the trial. | The plan subscription bills on its cadence. |
| Bitfield is installed somewhere but never opened or triggered in the billing window. | That dormant install does not create a runtime/device unit by itself. |
| Bitfield is triggered once on your laptop this month. | That laptop uses 1 runtime/device unit for the window. |
| The same laptop uses Bitfield 1,000,000 times this month. | Still the same runtime/device unit, not 1,000,000 units. |
| Bitfield is triggered on laptop, phone, and one server this month. | 3 runtime/device units. |
| 100,000 people view a static public page. | Those visitors are not runtime/device units. |
| A package folder sits in a repo and nobody runs it. | Nothing new is counted. |

So the charging model is not "installed forever." It is not "runtime seconds." It is not "visitors." It is triggered Bitfield runtime usage, counted by runtime identity during the billing window.

## What "triggered" means

A trigger is the moment Bitfield has to use the protected runtime path for a real runtime identity. In customer language: the device, server, or runtime environment asks Bitfield for the runtime permission it needs to run.

When that succeeds, Bitfield records the signed fact that billing is allowed to count. Billing reads those facts. It does not guess from route names, page labels, marketing pages, package names, or how many people looked at a page.

## Do not mix up these two things

| Thing | Plain meaning |
|---|---|
| Plan subscription | The base plan you chose after the trial. |
| Runtime/device unit | A triggered Bitfield runtime identity inside the billing window. |

If you keep a paid plan, the plan subscription still exists. The trigger rule explains what consumes the included runtime/device units and what does not.

## What counts

| Thing | Counts when |
|---|---|
| Laptop or desktop | Bitfield is triggered there during the billing window. |
| Phone or tablet | Bitfield is triggered there during the billing window. |
| Server or worker | Bitfield is triggered there during the billing window. |
| Local box, kiosk, or test machine | Bitfield is triggered there during the billing window. |
| Customer device | The customer's environment triggers Bitfield runtime access. |

The plain test is: did this runtime identity actually trigger Bitfield during this billing window?

## What does not count by itself

| Thing | Why |
|---|---|
| Dormant install | Files sitting there are not a trigger. |
| Package folder | Product material is not runtime usage. |
| Project | A project is not the billing unit. |
| Static page view | Reading static output does not trigger Bitfield runtime. |
| Website visitor | A visitor is not a runtime identity unless their environment actually runs Bitfield. |
| Click count | Pricing is not per click. |
| Runtime duration | Pricing is not per second or per minute. |
| Revenue | Pricing is not a revenue share. |

## Common scenarios

### Installed but not used

You installed Bitfield on a laptop months ago. This month you never open it, start it, or trigger the runtime path.

Result: your active plan still follows its subscription cadence, but that dormant laptop does not create a new runtime/device unit by itself.

### Same machine, heavy use

You trigger Bitfield on your laptop once, then use the product all month.

Result: that laptop is one triggered runtime/device unit for the billing window. It is not per action, per click, or per second.

### Local app on customer machines

You ship a Bitfield-powered local app to 100 customer machines, and each one triggers Bitfield in the billing window.

Result: those 100 triggered runtime identities are the count.

### Hosted product

You run a hosted product from one server, and that server triggers Bitfield runtime access.

Result: the server runtime identity is what gets counted. Visitors do not become separate runtime/device units just by viewing output from that server.

### Static public page

You publish a landing page as ordinary static or hosted output.

Result: visitors are not counted. The page can get traffic without creating runtime/device units for every person who reads it.

## How to avoid surprise bills

1. Use Bitfield where you need Bitfield runtime behavior.
2. Keep static marketing pages static when they only need copy, images, links, and forms.
3. Do not trigger Runtime Kit just to display public text or images.
4. Revoke old devices, servers, and test machines that should not be used anymore.
5. Do not put a Bitfield key into public browser code.
6. Check the account portal before rolling out to a fleet, team, or customer set.

## Common failures

| Symptom | Cause | Fix |
|---|---|---|
| Someone thinks a dormant install is charged forever | Install and trigger got mixed together | Say files sitting there are not a trigger |
| Someone thinks every visitor is charged | Static traffic and runtime access got mixed together | Keep static pages static when they do not need Runtime Kit |
| Someone thinks heavy use becomes per-click billing | Trigger count and usage amount got mixed together | Say the same runtime identity is not counted once per click |
| Someone puts a key in public browser code | Public page and runtime identity were not separated | Keep keys in the activation/runtime path, not public page code |

Manage account access and runtime identities from [account.bitfield.so](https://account.bitfield.so).
