## Use Runtime Kit here

| Use Runtime Kit when the surface needs... | Why |
|---|---|
| Named Bitfield data | The surface reads data Runtime Kit prepares by name. |
| Target requests | A button or action asks a Bitfield target to do work. |
| Package boundaries | Product material belongs to packages instead of tangled app imports. |
| Local state | The product needs Bitfield state on the device or runtime environment. |
| Offline-capable behavior | The product should keep useful state close to where it runs. |
| Replaceable product pieces | Packages can change without rewriting every screen. |

That is real Runtime Kit work.

## Do not trigger Runtime Kit just for this

| Public page need | Better shape |
|---|---|
| Static headline and body copy | Static or ordinary hosted output. |
| Marketing images | Static image assets. |
| A pricing page that only explains plans | Static or ordinary hosted output. |
| A docs page | Static docs output. |
| A public blog post | Static or ordinary hosted output. |
| A button that links somewhere | Ordinary link or form behavior. |

This is not anti-Bitfield. It is cost-aware Bitfield. Use the runtime where the runtime is doing something valuable.

## Start with the symptom

| Symptom | What probably happened | First fix |
|---|---|---|
| A public page feels like it will create usage | Runtime Kit was added to static content | Keep the page static unless it needs live Bitfield data or target requests |
| Visitors are being discussed like devices | Traffic and runtime identities were mixed together | Ask which environment actually triggers Bitfield |
| An AI agent adds Runtime Kit to every page | The instruction did not separate marketing and product surfaces | Use the prompt below before the agent writes code |
| A test machine keeps showing up | Old runtime identity was not cleaned up | Revoke old devices and test machines in the account portal |

## Three shapes

### Static public page

```text
build or export step
  -> HTML/CSS/JS/images
  -> web host serves files
  -> visitors read the page
```

Result: visitors do not trigger Bitfield runtime usage.

The build machine can still be a Bitfield runtime identity if the build itself triggers Bitfield. The public visitor reading static output does not become one.

### Hosted Runtime Kit product

```text
server triggers Bitfield runtime access
  -> product surface uses Runtime Kit
  -> visitors use the hosted product
```

Result: the server runtime identity is the counted runtime/device unit. Visitors are not counted separately unless their own environment also triggers Bitfield runtime access.

### Local or downloaded product

```text
customer device triggers Bitfield runtime access
  -> Runtime Kit talks to local Bitfield state
  -> product runs on that machine
```

Result: each triggered customer runtime identity can consume a runtime/device unit for the billing window.

## Mixed site example

`acme.com` can have both public pages and a Runtime Kit product.

| URL | Good shape | Billing meaning |
|---|---|---|
| `/` | Static landing page | Visitors do not trigger runtime usage. |
| `/pricing` | Static pricing page | Visitors do not trigger runtime usage. |
| `/docs` | Static docs output | Visitors do not trigger runtime usage. |
| `/app` | Runtime Kit product surface | The device, server, or environment that triggers Bitfield runtime access can be counted. |

The path name does not decide billing. The trigger does.

## What to tell an AI agent

Use this when an AI agent is building a public site around Bitfield:

```text
Build public marketing pages as static or ordinary hosted output unless the page needs live Bitfield data or target requests.
Use Runtime Kit only for surfaces that need named data, target requests, local Bitfield state, package runtime behavior, or offline-capable product behavior.
Never put a Bitfield key into public browser code.
Do not trigger Runtime Kit just to render static copy, images, links, or a brochure page.
```

## Launch checklist

Before launch, answer these plainly:

1. Which devices, servers, or runtime environments will trigger Bitfield?
2. Which public pages are just static output?
3. Which product surfaces really need Runtime Kit?
4. Are any Bitfield keys or activation files exposed to public browser code?
5. Are old test machines and dead runtime identities revoked in the account portal?

The bill should follow the first answer, not the number of people who read public pages.

Do not copy one device's local activation folder to another device. Activate the runtime identity properly from the account path.

## Verify

The page is wired correctly when you can point to each public URL and answer one question: does this URL need to trigger Bitfield runtime access?

Static pages should load without a Bitfield key in public browser code. Runtime Kit product surfaces should use the public Runtime Kit calls and account-owned activation path. The component renders real data only when the named data exists and the runtime path is actually needed.
