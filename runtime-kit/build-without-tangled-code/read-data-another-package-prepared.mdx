<div className="bf-article">

<p className="bf-lead">
When one package prepares data and another package needs it, the consumer should read a prepared input. It should not import the producer package, parse its files, or guess the storage address.
</p>

Product scene: a project-preview package prepares a list of preview URLs and statuses. A web-frame package wants to render the active URL. The web-frame should not know how project-preview stores those records. It should read the view it was given.

Expected result: the producing package can change how it prepares records while the consuming package keeps reading the same public input shape.

## Traditional app shape

```ts
import { getPreviewRecords } from '../project-preview/store';

export function WebFrame() {
  const preview = getPreviewRecords().find(item => item.active);
  return renderUrl(preview.url);
}
```

This ties the web frame to the project-preview package. The consumer now knows the producer's module path, function name, record shape, and selection logic.

## Bitfield shape

```text
producer package prepares records
Runtime Kit exposes a prepared input
consumer package reads that input
consumer decodes only the public view it asked for
```

The consumer can read `source-by-label`, `source-by-address`, `selected-file`, `current-project`, or any other input that the package boundary exposes. It does not read producer private code.

## Code translation

Traditional:

```ts
import { previewStore } from '../project-preview/store';

const active = previewStore.getActivePreview();
```

Bitfield:

```ts
const active = readInput('project-preview-surface');
```

React adapter example:

```tsx
import { useBitfieldData } from '@bitfield/runtime-kit/react';

type PreviewSurface = {
  projectId: string;
  name?: string;
  url?: string;
  isConnected?: boolean;
  status?: string;
};

export function PreviewPanel() {
  const preview = useBitfieldData<PreviewSurface>('project-preview-surface');

  if (preview.loading) return <p>Loading preview</p>;
  if (preview.error) return <p>Could not load preview.</p>;
  if (!preview.data?.url) return <p>No preview URL yet.</p>;

  return <iframe title={preview.data.name ?? 'Project preview'} src={preview.data.url} />;
}
```

## Descriptor-projected records

Some consumers need a different view of the same source. That should be data, not a new import path.

```text
source-by-label:
  params.source_address_label = project-preview-surface
  field map says which public field is the URL
```

This lets the consumer read a projected view. It does not make the consumer learn the producer's private data layout.

## What this prevents

| Bad shape | Why it breaks | Better shape |
|---|---|---|
| Consumer imports producer package. | Moving the producer breaks the consumer. | Consumer reads a prepared input. |
| Consumer parses `things-to-store-and-run.json`. | UI code becomes package-loader code. | Runtime Kit resolves the package boundary. |
| Consumer builds storage addresses by string concatenation. | Address rules leak into app code. | Consumer uses a named input. |
| Consumer assumes producer field names forever. | Producer cannot change its private shape safely. | Descriptor data maps the public view. |

React is one adapter for reading a prepared input. A Swift, Kotlin, terminal, or future shell should follow the same boundary: read the named input, not the package that prepared it.

## AI agent should know

If generated code says `import ... from '../other-package'`, ask why. A consumer package should receive a prepared input or a target name. It should not inspect another package's source files.
