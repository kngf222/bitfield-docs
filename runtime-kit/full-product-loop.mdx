# Full product loop.

<p className="bf-lead">
This page shows the real shape: packages provide reads and targets, placeable surfaces declare where UI can appear, and the shell stays dumb enough to rearrange the product without knowing product meaning.
</p>

## What this is

One package and one screen proves the read/request boundary. A product needs one more layer: a generic arrangement shell.

The shell should not know what a launch dashboard is. It should not know what support means. It should not know which package owns help search.

The shell should only know this:

| Generic thing | What it means |
|---|---|
| Surface descriptor | A piece of UI that can be placed in a region. |
| Region | A named area such as `main`, `rail`, or `panel`. |
| Prepared input names | Data names the surface is allowed to read. |
| Target names | Work names the surface is allowed to request. |
| Component key | The renderable body for that surface. |

That is the same class of idea as a command center: the arrangement model is generic, the sidebar is generic, and the product meaning lives behind descriptors.

## The product map

```text
Package set
  -> packages
  -> prepared inputs
  -> targets
  -> surface descriptors

Product shell
  -> reads surface descriptors
  -> renders a generic sidebar from labels and order
  -> renders the active surface in a generic region

Surface body
  -> reads only the prepared inputs named by its descriptor
  -> calls only the targets named by its descriptor
```

The product becomes larger without making the shell import package setup code or product-specific screen names.

## Placeable surface descriptors

A placeable surface is a small product contract. It tells the shell what can be placed, where it belongs, and which Runtime Kit names its body can use.

```ts
type SurfaceRegion = 'main' | 'rail' | 'panel';

type PlaceableSurface = {
  id: string;
  label: string;
  region: SurfaceRegion;
  order: number;
  componentKey: string;
  preparedInputs: string[];
  targets: string[];
};

export const launchSurfaces: PlaceableSurface[] = [
  {
    id: 'launch.home',
    label: 'Launch',
    region: 'main',
    order: 10,
    componentKey: 'surface.launch.home',
    preparedInputs: ['welcome-copy', 'launch-checklist'],
    targets: ['launch.next-step'],
  },
  {
    id: 'launch.help',
    label: 'Help',
    region: 'panel',
    order: 20,
    componentKey: 'surface.launch.help',
    preparedInputs: [],
    targets: ['help.search'],
  },
  {
    id: 'launch.support',
    label: 'Support',
    region: 'panel',
    order: 30,
    componentKey: 'surface.launch.support',
    preparedInputs: [],
    targets: [],
  },
];
```

Notice what is missing: no dashboard branch, no help branch, no support branch in the shell. The product-specific names are data on the descriptor.

## Dumb shell

The shell receives surface descriptors and a render registry. It builds navigation from descriptor labels and chooses the active surface by id.

```tsx
import { useState } from 'react';

type SurfaceComponentProps = {
  surface: PlaceableSurface;
};

type SurfaceRegistry = Record<string, React.ComponentType<SurfaceComponentProps>>;

type ProductShellProps = {
  surfaces: PlaceableSurface[];
  registry: SurfaceRegistry;
};

export function ProductShell({ surfaces, registry }: ProductShellProps) {
  const ordered = [...surfaces].sort((a, b) => a.order - b.order);
  const [activeSurfaceId, setActiveSurfaceId] = useState(ordered[0]?.id ?? '');
  const activeSurface = ordered.find((surface) => surface.id === activeSurfaceId) ?? ordered[0];
  const ActiveSurface = activeSurface ? registry[activeSurface.componentKey] : null;

  return (
    <div className="product-shell">
      <aside aria-label="Product surfaces">
        {ordered.map((surface) => (
          <button
            key={surface.id}
            type="button"
            aria-current={surface.id === activeSurface?.id ? 'page' : undefined}
            onClick={() => setActiveSurfaceId(surface.id)}
          >
            {surface.label}
          </button>
        ))}
      </aside>

      <main data-region={activeSurface?.region ?? 'main'}>
        {activeSurface && ActiveSurface ? <ActiveSurface surface={activeSurface} /> : null}
      </main>
    </div>
  );
}
```

That shell can place any future surface that follows the descriptor. It does not change when you add analytics, onboarding, billing, settings, or a new customer-facing workflow.

## Surface bodies

The surface body is where product work happens. It reads Runtime Kit data and calls Runtime Kit targets.

```tsx
import { sendRequestToBitfieldTarget } from '@bitfield/runtime-kit';
import { useBitfieldData } from '@bitfield/runtime-kit/react';

type WelcomeCopy = {
  headline: string;
  body: string;
};

type Checklist = {
  items: Array<{ id: string; label: string; done: boolean }>;
};

type NextStepReply = {
  nextStep: string;
  reason: string;
};

export function LaunchHomeSurface({ surface }: SurfaceComponentProps) {
  const welcome = useBitfieldData<WelcomeCopy>('welcome-copy');
  const checklist = useBitfieldData<Checklist>('launch-checklist');

  async function getNextStep(): Promise<NextStepReply> {
    const reply = await sendRequestToBitfieldTarget({
      target: 'launch.next-step',
      payload: {
        surfaceId: surface.id,
        completed: checklist.data?.items.filter((item) => item.done).map((item) => item.id) ?? [],
      },
    });

    return JSON.parse(new TextDecoder().decode(reply.payload)) as NextStepReply;
  }

  if (welcome.loading || checklist.loading) return <section>Loading.</section>;
  if (welcome.error || checklist.error) return <section>Could not load this surface.</section>;
  if (!welcome.data || !checklist.data) return <section>No data yet.</section>;

  return (
    <section>
      <h2>{welcome.data.headline}</h2>
      <p>{welcome.data.body}</p>

      {checklist.data.items.map((item) => (
        <p key={item.id}>{item.done ? 'Done' : 'Next'}: {item.label}</p>
      ))}

      <button type="button" onClick={getNextStep}>
        Show next step
      </button>
    </section>
  );
}
```

This surface can be moved from `main` to `panel`, opened beside another surface, or reused in a different shell. The product work stays with the surface body. The arrangement stays with the shell.

## A second surface using the same request primitive

```tsx
import { useRef } from 'react';
import { sendRequestToBitfieldTarget } from '@bitfield/runtime-kit';

type HelpSearchReply = {
  results: Array<{ title: string; excerpt: string }>;
};

export function LaunchHelpSurface() {
  const current = useRef<AbortController | null>(null);

  async function search(query: string): Promise<HelpSearchReply> {
    current.current?.abort();
    current.current = new AbortController();

    const reply = await sendRequestToBitfieldTarget(
      { target: 'help.search', payload: { query } },
      current.current.signal,
    );

    return JSON.parse(new TextDecoder().decode(reply.payload)) as HelpSearchReply;
  }

  return <button type="button" onClick={() => search('activation')}>Search help</button>;
}
```

The app has another capability. The shell still did not change.

## Support surface

The support surface should show the public contract that helps someone debug the product loop.

```tsx
type RuntimeKitContract = {
  packageSet: string;
  packages: string[];
  preparedInputs: string[];
  targets: string[];
  surfaces: Array<{ id: string; label: string; region: string }>;
};

export function LaunchSupportSurface({ contract }: { contract: RuntimeKitContract }) {
  return (
    <section>
      <h2>Runtime Kit contract</h2>
      <p>Package set: {contract.packageSet}</p>
      <p>Packages: {contract.packages.join(', ')}</p>
      <p>Prepared inputs: {contract.preparedInputs.join(', ')}</p>
      <p>Targets: {contract.targets.join(', ')}</p>
      <p>Surfaces: {contract.surfaces.map((surface) => surface.id).join(', ')}</p>
    </section>
  );
}
```

That gives support enough information to debug the public chain:

```text
Package set: launch-product
Packages: launch-product
Prepared inputs: welcome-copy, launch-checklist
Targets: launch.next-step, help.search
Surfaces: launch.home, launch.help, launch.support
```

It does not ask the user to paste local state contents, activation files, keys, or package implementation files.

## Register the product

The product assembly step wires descriptors to renderable bodies. This is the only place product names meet the generic shell.

```tsx
const surfaceRegistry: SurfaceRegistry = {
  'surface.launch.home': LaunchHomeSurface,
  'surface.launch.help': LaunchHelpSurface,
  'surface.launch.support': LaunchSupportSurface,
};

export function LaunchProduct() {
  return <ProductShell surfaces={launchSurfaces} registry={surfaceRegistry} />;
}
```

That line is not the architecture. It is the product assembly. The architecture is the descriptor shape plus the shell that can place descriptors without knowing the product.

## Scale the same shape

Now add an analytics package without touching the shell:

```ts
export const analyticsSurfaces: PlaceableSurface[] = [
  {
    id: 'analytics.summary',
    label: 'Summary',
    region: 'main',
    order: 10,
    componentKey: 'surface.analytics.summary',
    preparedInputs: ['daily-summary'],
    targets: ['analytics.explain'],
  },
  {
    id: 'analytics.anomalies',
    label: 'Anomalies',
    region: 'panel',
    order: 20,
    componentKey: 'surface.analytics.anomalies',
    preparedInputs: ['anomaly-list'],
    targets: ['analytics.explain', 'analytics.resolve'],
  },
];
```

Then merge descriptors:

```tsx
export function FounderWorkspace() {
  return (
    <ProductShell
      surfaces={[...launchSurfaces, ...analyticsSurfaces]}
      registry={{
        ...launchSurfaceRegistry,
        ...analyticsSurfaceRegistry,
      }}
    />
  );
}
```

The product now has launch, help, support, summary, and anomalies. The shell still renders labels, active ids, regions, and components. It does not know product meaning.

## Review the full loop

Before you call the feature done, write this checklist beside it:

```text
Package set:
  launch-product

Prepared inputs:
  welcome-copy
  launch-checklist

Targets:
  launch.next-step
  help.search

Placeable surfaces:
  launch.home -> main -> welcome-copy, launch-checklist -> launch.next-step
  launch.help -> panel -> help.search
  launch.support -> panel -> public contract only

Generic shell proof:
  The sidebar renders from surface labels.
  The active region renders from surface.region.
  The render body resolves from surface.componentKey.
  Adding a new surface does not edit shell code.

Public request/reply contracts:
  launch.next-step -> { surfaceId: string; completed: string[] } -> { nextStep: string; reason: string }
  help.search -> { query: string } -> { results: Array<{ title: string; excerpt: string }> }
```

If an AI agent built the feature, make it return this checklist before you accept the code. If a teammate built it, use the same checklist in review.
