# How Bitfield is this fast.

<p className="bf-lead">
Persistent does not mean every read waits for storage hardware. Local does not mean fake. Fast claims only mean something when the measured category is visible.
</p>

## The short version

Bitfield is fast because the storage shape removes work from the hot path. The source of truth is durable. The warm local read path is fast because the operating system already has the needed bytes available and Bitfield can read by the shape it stored.

## Persistent does not mean every read waits for disk

A memory map connects a durable file to the program's address space. The file is still the durable thing. When the operating system already has the needed pages warm, reading those bytes avoids the normal database request path.

If the page is cold and hardware has to fetch it, physics is slower. That is why the honest wording is <span className="bf-blue">persistent source of truth with a warm local fast path</span>.

## Content addresses remove searching

Most databases start from an application question and do work to answer it: parse, plan, traverse indexes, check structures, and move data through a database process.

Bitfield stores content by identity. Once it knows the exact place to read, the H0 path is almost just: go to these bytes.

## No benchmark gets to hide its category

This is not a promise that a cold disk read takes less than a nanosecond. It is not a claim that a network round trip disappears. It is not saying every vendor benchmark is comparable to every other one.

<div className="bf-callout">
The rule is simple: show the mechanism, show the tradeoff, and say what work is actually being measured.
</div>
